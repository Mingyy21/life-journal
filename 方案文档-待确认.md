# 人生手记 — 架构重构与功能完善方案

## 一、项目全景诊断

### 当前架构

```
RootLayout
  └─ AppProviders (AuthProvider)
      └─ Layout (导航)
          └─ Page (每个页面独立加载)
              ├─ useEffect → ensureDb()
              │   ├─ db.open()        → 等 auth session（最长8s）
              │   ├─ initDefaultData() → 13次单独 GET（5领域+8课题）
              │   └─ resolve
              ├─ db.topics.toArray()      → GET /topics?userId=xxx
              ├─ db.diaries.toArray()     → GET /diaries?userId=xxx
              ├─ db.events.toArray()      → GET /events?userId=xxx
              └─ ...更多数据
```

### 根源问题（6个）

| # | 问题 | 表现 | 根因 |
|---|------|------|------|
| **R1** | 无全局数据共享 | 6个页面各自加载 topics/domains，同一 session 内重复 6+ 次 | 架构缺少数据层 |
| **R2** | Auth 阻塞渲染流水线 | 每个页面 mount 后才开始等 auth，等完才开始 fetch | `db.open()` 同步阻塞设计 |
| **R3** | 初始化做 13 次独立查询 | `initDefaultData()` 逐个 `db.topics.get(id)` 检查存在性 | Code smell 遗留 |
| **R4** | Dexie-compat 层低效 | 全表 `SELECT *` 取所有列，客户端 JS 排序，无字段裁剪 | 适配器模式固有开销 |
| **R5** | 无请求去重 | 同一页面多个组件调用 `db.topics.toArray()` 产生重复请求 | 无缓存层 |
| **R6** | 无分页 | 日记列表永远 `SELECT *` 全量拉取，页面上只显示前 50 条 | Dexie API 天然全量 |

### 连带影响

- Trend Report 在客户端调用 DeepSeek → 读不到 `process.env.DEEPSEEK_API_KEY` → 永远失败
- 批量写操作后全量重新 fetch → 3-5s 等待
- 数据依赖关系没有声明式管理 → 组件不知道数据何时更新

## 二、目标架构

```
RootLayout
  └─ AppProviders
      ├─ AuthProvider           ← 非阻塞，推模式
      ├─ QueryClientProvider    ← 新增：@tanstack/react-query
      └─ InitProvider           ← 新增：一次性全局初始化
          └─ Layout
              └─ Pages
                  ├─ useQuery(['topics'], { staleTime: 300000 })   → 首次加载后缓存5分钟
                  ├─ useQuery(['domains'], { staleTime: 300000 })  → 同上
                  ├─ useQuery(['diaries', filter], { staleTime: 0 })→ 页面级最新
                  ├─ useQuery(['events'])                           → 按需缓存
                  └─ useMutation(+ invalidate)                      → 写后自动刷新
```

### 改了什么

| 变更 | 之前 | 之后 |
|------|------|------|
| **数据获取** | 每个页面 `useEffect` + 手动 setState | `useQuery` 声明式，自动缓存/去重 |
| **引用数据** | 每个页面拉一次 topics/domains | `staleTime: 5min`，全局共享 |
| **Auth 阻塞** | `db.open()` 等8秒 | AuthProvider 初始化时不阻塞渲染 |
| **初始化** | `initDefaultData()` 13次 GET | 首次初始化后标记已完成，后续跳过 |
| **请求** | `SELECT *` 全列全量 | 按需 `select()` 裁剪字段 |
| **分页** | 无 | 日记列表 `range(0, 49)` |
| **写操作** | 写完后手动 reload | `useMutation` + `invalidateQueries` 自动刷新 |
| **去重** | 无 | React Query 自动合并相同 key 请求 |

## 三、方案详解

### 3.1 安装依赖

```bash
npm install @tanstack/react-query
```

`@tanstack/react-query` v5，树摇优化，gzip ~12KB。这是唯一新增的运行时依赖。

### 3.2 React Query QueryKey 设计

统一的数据标识符体系，确保缓存一致性：

```typescript
// queryKey 工厂
const queryKeys = {
  topics:      ['topics'] as const,
  domains:     ['domains'] as const,
  diaries:     (filter?: DiaryFilter) => ['diaries', filter] as const,
  events:      (topicId?: string) => ['events', topicId] as const,
  event:       (id: string) => ['event', id] as const,
  insights:    (filter?: InsightFilter) => ['insights', filter] as const,
  analyses:    (diaryId: string) => ['analysis', diaryId] as const,
  stats:       { weekly: ['stats', 'weekly'] as const, monthly: ['stats', 'monthly'] as const },
};
```

缓存规则：

| QueryKey | staleTime | gcTime | 说明 |
|----------|-----------|--------|------|
| `['topics']` | 5 min | 30 min | 引用数据，几乎不变 |
| `['domains']` | 5 min | 30 min | 同上 |
| `['diaries', filter]` | 0 | 5 min | 每次进页面都刷新 |
| `['events', topicId]` | 1 min | 10 min | 事件状态可能变化 |
| `['event', id]` | 30 s | 10 min | 单个事件 |
| `['insights']` | 1 min | 10 min | 感悟列表 |

### 3.3 Auth 重写 — 非阻塞初始化

**当前问题**：`AuthProvider` → `getSession()` 发生在客户端渲染后，`setUserId()` → 通知 `db.open()` 的 Promise resolve → 页面才能开始 fetch。

**改造**：AuthProvider 一开始就给一个 `initializing: true` 状态，页面可以在加载状态中渲染。同时 App 级别启动一次初始化。

```typescript
// AuthProvider 中
const [state, setState] = useState<AuthState>({
  initializing: true,  // ← 新增初始状态
  userId: null,
  email: null,
});

useEffect(() => {
  supabase.auth.getSession().then(({ data }) => {
    setState({
      initializing: false,
      userId: data.session?.user?.id ?? null,
      email: data.session?.user?.email ?? null,
    });
    // 不再立即 setUserId → 不再阻塞
  });
}, []);
```

### 3.4 InitProvider — 一次初始化，全局共享

新增 `InitProvider`，在 AuthProvider 之后，Layout 之前。功能：

1. 调用 `ensureDb()` 一次（包含 initDefaultData）
2. `initDefaultData` 改为批量检查（`db.topics.toArray()` 一次拿到全部，而不是 8 次 get）
3. 初始化完成后设置 `ready` 状态
4. 所有页面通过 `useInit()` 获取 ready 状态，无需再调用 `ensureDb()`

```typescript
// 重写 initDefaultData
export async function initDefaultData(): Promise<void> {
  const [existingTopics, existingDomains] = await Promise.all([
    db.topics.toArray(),     // 1 次查询拿到所有课题
    db.lifeDomains.toArray(), // 1 次查询拿到所有领域
  ]);
  const existingTopicIds = new Set(existingTopics.map(t => t.id));
  const existingDomainIds = new Set(existingDomains.map(d => d.id));
  
  const toAddTopics = DEFAULT_TOPICS.filter(t => !existingTopicIds.has(t.id));
  const toAddDomains = DEFAULT_LIFE_DOMAINS.filter(d => !existingDomainIds.has(d.id));
  
  if (toAddTopics.length > 0) await db.topics.bulkAdd(toAddTopics);
  if (toAddDomains.length > 0) await db.lifeDomains.bulkAdd(toAddDomains);
}
// 改动：13次 GET → 2次 GET + 批量写入
```

### 3.5 Dexie-compat 层优化

| 优化项 | 当前 | 改造后 |
|--------|------|--------|
| 字段选择 | `SELECT *` | 支持 `select(col1, col2)` |
| 排序 | 客户端 JS 排序 | 利用 Supabase `.order()` 服务端排序（已在 buildQuery 中有，但 toArray 的兜底 JS sort 可移除） |
| 分页 | 无 | 新增 `limit(n)` / `offset(n)` |
| 批量操作 | `bulkAdd` 逐条 insert | 使用 Supabase `.insert(rows)` 批量（已有） |

### 3.6 趋势报告 API Route

新建 `src/app/api/trend/route.ts`，把 DeepSeek 调用放到服务端：

```
客户端 (stats/page.tsx)
  → POST /api/trend { scope, diaryIds, ... }
  → API Route (服务端)
    → process.env.DEEPSEEK_API_KEY (可用)
    → call DeepSeek
  → return 结果
  → React Query 缓存
```

### 3.7 统计页图表修复

**写作频率**：添加 `min-height` 和显式数字标签

```tsx
// 之前：柱子只有 background-color，无标签
// 之后：每个柱子上方显示篇数，hover 显示日期，数据不足时显示文字
{freqEntries.slice(-60).map(([date, count]) => (
  <div key={date} className="flex flex-col items-center" title={`${date}: ${count}篇`}>
    <span className="text-[10px] text-calm-400 mb-0.5">{count}</span>
    <div className="w-full rounded-t-sm bg-primary-400/60 hover:bg-primary-400"
      style={{ height: maxFreq > 1 ? `${(count / maxFreq) * 100}%` : '100%', minHeight: 4 }} />
  </div>
))}
```

**情绪趋势**：即使无 AI 分析，用规则引擎初步的情绪标注做降级显示。

### 3.8 编辑器保存按钮（移动端修复）

使用 `visualViewport` API 检测键盘弹出：

```typescript
useEffect(() => {
  const handler = () => {
    if (window.visualViewport) {
      // 键盘弹出时，将底部栏推到 visualViewport 底部
      const diff = window.innerHeight - window.visualViewport.height;
      if (diff > 100) { // 键盘大概率弹出了
        setKeyboardOpen(true);
      } else {
        setKeyboardOpen(false);
      }
    }
  };
  window.visualViewport?.addEventListener('resize', handler);
  return () => window.visualViewport?.removeEventListener('resize', handler);
}, []);
```

## 五、阶段依赖关系

```
阶段一（数据架构）
  │
  ├── 阶段一测试通过 ────┐
  │                     │
  ▼                     ▼
阶段二（功能修复）     （阶段二依赖阶段一的数据层）
  │
  ├── 阶段二测试通过 ────┐
  │                     │
  ▼                     ▼
阶段三（感悟标记）     （阶段三依赖阶段一的数据层，
                        依赖阶段二的编辑器修复）
```

**规则**：
1. 每个阶段完成后，运行完整测试清单
2. 测试未通过则修复，最多重试 3 次
3. 阶段三必须在阶段一和阶段二都通过后才能开始
4. 每个阶段完成后 `git commit`，确保回退点
